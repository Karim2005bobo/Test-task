Техническое задание (БА+СА) для аналитика

Задача 1

Создайте модель в нотации BPMN 2.0 на основе приведенного ниже описания рабочего процесса.

Описание: в компании «X» выдача ИТ-оборудования пользователям осуществляется по заявке, которую необходимо оформить в системе № 1. Пользователь офиса при необходимости подает заявку на оборудование через почту или по телефону в первую линию технической поддержки. Он получает номер обращения, по которому может отследить, что происходит с его заявкой. Первая линия технической поддержки оформляет заявку на оборудование (на основании заявки пользователя) в системе № 2 для отдела снабжения. Отдел снабжения принимает заявку в системе № 2. Отдел снабжения видит, что есть заявка на оборудование, по уведомлению в почте, а также на экране своего личного кабинета в системе № 2. Затем отдел снабжения проверяет наличие запрашиваемого пользователем оборудования в системе № 3. Если оборудование есть, отдел снабжения связывается с отделом логистики и отправляет оборудование в первую линию технической поддержки. Техническая поддержка выдаёт это оборудование пользователю и закрывает заявку пользователя в системе № 1 и в системе № 2. Пользователь получает оборудование и может выставить оценку работе технической поддержки. Если оборудования нет, отдел снабжения отправляет поставщику заявку на закупку по электронной почте. Ждёт оборудование. Поставщик привозит его. Затем отдел снабжения принимает оборудование в своей системе № 3. Далее вызывает логистов и передаёт оборудование в техподдержку. Техподдержка выдаёт оборудование пользователю и закрывает заявку пользователя в системе № 1 и заявку в системе № 2. Пользователь получает оборудование и может выставить оценку техподдержке. На модели можно указать вопросы и противоречия в описании рабочего процесса. (Чего не хватает для грамотного отражения модели «как есть»)

Задача 2 Представьте, что вы аналитик личного кабинета продавца. Вам поручили сформулировать требования для реализации новой функции (функционала), которая позволит публиковать товары на маркетплейсе (витрине товаров). Перед вами стоят следующие задачи: составить высокоуровневую постановку в формате пользовательской истории и описать требования к новому функционалу в формате вариантов использования (Use Case). Дополнительно: нарисовать диаграмму процесса в удобной нотации.

Задача 3 На картинке (скачать можно по ссылке — https://drive.google.com/file/d/1fi9JLRvO6bhS1uJJ2PUX8ZTOq0Eq_At0/view?usp=drive_link) изображен интерфейс регистрации пользователя (как успешный экран создания пользователя, так и варианты с ошибками). Задание 3.1: Требуется описать REST API-интерфейс на стороне бэк-сервиса, который должен вызываться фронтендом при нажатии на кнопку «Зарегистрироваться». Описание должно включать: http-метод + URL входные параметры (название, тип, обязательность, ограничения) выходные параметры при успешном ответе (название, тип, обязательность, ограничения) выходные параметры при ответе с ошибкой (название, тип, обязательность, ограничения) описание ошибок — коды ответов (главное — правильно указать, клиентские или серверные) + текст сообщения пример запроса пример ответа (для успешного сценария и сценария с ошибкой). Требований к формату нет (может быть в виде таблицы или просто текстом). Допускается описание в формате спецификации openAPI. Полученный YAML-файл приложить в виде текста к решению или в виде отдельного файла (не забудьте открыть доступ).

Задание 3.2 Описать подробный пошаговый алгоритм, создания Пользователя на стороне бэк сервиса при вызове, получившегося в Задачи 3.1 метода. Алгоритм должен подробно описывать какие проверки требуется выполнять бэк сервису после получения запроса на создание нового Пользователя (при нажатии кнопки “Register”), прежде чем создать Пользователя в БД. Описание может быть в виде текста или диаграммы (например, activity)/блок-схемы.
